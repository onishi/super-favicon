# SUPER-FAVICON

Favicon を動的に書き換えることで、Favicon の画像を 32x32 ピクセルのゲーム画面に見立てて遊べるミニゲーム集。

## コンセプト

- ブラウザタブの Favicon（32x32px）を Canvas で描画 → 画像化し、一定間隔で `<link rel="icon">` を差し替えることでアニメーションさせる
- 「タブの中で動くゲーム」という制約を逆手に取り、複数のレトロゲームを実装する
- ゲームロジック自体は 32x32 の2値 or 少数色のピクセルグリッドで完結させ、Favicon 描画と画面内表示（Mobile 用）の両方に同じデータを流し込む

## 対応環境

- PC（デスクトップブラウザ）/ Mobile（スマートフォン）両対応
- **PC**: Favicon が実際にアニメーションして見える
- **Mobile**: ブラウザ側で Favicon の変化が視認できないため、代わりにページ全体を包む擬似ブラウザUI（`BrowserChrome`）のタブ内にFaviconを縮小表示する（Favicon への書き込み自体は継続する）
  - タブ内の縮小表示では遊びにくいため、擬似ブラウザUIの信号機風ボタン（緑）を押すと大きいゲーム画面を表示できる（デフォルトは非表示）

## ページ全体のブラウザ風UI

「Favicon がゲーム画面」というコンセプトをページ内でも常に体感できるよう、アプリ全体を簡易的なブラウザウィンドウ風の見た目（`BrowserChrome`）で包む。

- 画面上部にタブバー（信号機風の3つのドット＋タブ1枚）とアドレスバー風の帯を常時表示する
- タブには実際の `<link rel="icon">` の内容とタブタイトル（`document.title`）をポーリングして反映し、実物のブラウザタブと同じ内容をページ内でも常に確認できるようにする
  - これにより Mobile など実際のタブ変化が見えない環境でも、擬似的なタブでゲーム名・スコア・Faviconの見た目を確認できる
- 信号機風の3ドットは実際のウィンドウ操作風に機能する。緑（最大化）を押すとゲーム画面を大きく表示し、赤（閉じる）・黄（しまう）を押すと非表示に戻る。デフォルトは非表示
- ライト/ダークモードいずれの配色にも追従する

## 操作方法

十字キー＋1ボタン（合計5入力）を共通の操作体系とし、各ゲームがこれをマッピングして使う。

| 入力 | PC | Mobile |
|---|---|---|
| 上下左右 | カーソルキー or WASD | 画面上の十字ボタン |
| 決定/アクション | Space または Enter | 画面上の1ボタン |

- 入力は抽象化した「入力イベント（上/下/左/右/決定）」としてゲームロジックに渡し、キーボード/タッチのどちらから来たかをゲーム側が意識しなくてよい設計にする
- 画面上の十字＋1ボタンは、タップだけでなくキーボード操作時にも押下状態がハイライトされる（どちらの入力手段で操作しても同じ見た目でフィードバックする）
- ゲーム画面では PC は ESC キー、Mobile は画面上の十字＋1ボタンと並ぶ「RESET」ボタン（タップ）でタイトルへ戻れる
- タイトル画面にも同じ「RESET」ボタンを表示し、押すとカルーセルの選択位置をタイトルロゴ（先頭）まで戻す。PC では ESC キーが同じ操作に対応する

## Favicon 描画の仕組み（技術仕様）

- ゲーム状態は 32x32 のピクセルバッファ（各セルは on/off、またはカラーパレットのインデックス）として保持する
- 毎フレーム、このバッファを `<canvas width="32" height="32">` に描画する
- Canvas を `toDataURL()` 等で画像化し、`<link rel="icon">` の `href` を差し替えて Favicon を更新する
- ブラウザ・OS ごとに Favicon の更新頻度に制限がある場合があるため、更新間隔（フレームレート）は上限を設けて調整可能にする（目安: 数 FPS 〜 10FPS 程度から検証）
- Mobile など Favicon が視認できない環境では、同じ Canvas 描画結果を擬似ブラウザUIの最大化操作でページ内の `<canvas>` にも反映し、等倍 or 拡大表示する
- JS 側の描画が始まるまでの初期状態として、`public/favicon.svg`（32x32の黒一色）を静的な Favicon として用意する
- 32x32 という低解像度でもオブジェクトを視認しやすくするため、ゲーム内キャラクター（ON/ACCENT/ORANGE等の色付き値）の上下左右には自動でグロー効果（薄い色の添えドット）を付与する
  - `useGameRuntime` がゲームの `render()` 実行後に共通処理として適用するため、各ゲーム側は意識しなくてよい

## ドット絵エディタ

ゲーム作成・デバッグ補助ツールとして、Favicon 用ドット絵を作成できるエディタを実装する。隠し機能という位置づけのため、タイトル画面には案内を出さず、画面隅の小さなボタン（✎）からのみ開けるようにする。

- タブタイトルは「ドット絵エディタ」を表示し、ページ本体には見出しやメニューへの戻るボタンは表示しない（他画面と同様、タイトルはタブ側に一本化する）
- タイトル画面・ゲーム画面と同様、擬似ブラウザUIの信号機風ドット（赤・黄）を押すとメニューへ戻る。PC では ESC キーも同じ操作に対応する

- 32x32 ピクセルのグリッドをマウス（Mobile はタッチ）でドラッグして塗る/消す
- 描画内容はリアルタイムで実際の Favicon（`<link rel="icon">`）にも反映される。編集中の絵をそのままブラウザタブ・擬似ブラウザUIのタブで確認できる
- 描画エリアには8x8px単位のグリッド線をオーバーレイ表示し、セルの境界をわかりやすくする（実際の描画データには影響しない見た目だけの重ね描き）
- 各セルは RGB それぞれ4段階（0/85/170/255）の組み合わせによる64色パレット（8x8でキャンバス右に表示）から選んだ値で管理する
  - R=G=B=0（黒）は背景色と同じため、実質的に「オフ」として機能する
- ツールはペン・矩形（ドラッグでプレビューしながら塗る）・塗りつぶし（連続する同色領域を一括変更）の3種
- カーソル位置の色をパレット上にマーカー表示し、Shift + クリックでその色をスポイト（選択色に反映）できる。Shift 押下中はカーソルをクロスヘア（スポイトらしい見た目）に変更する
- 元に戻す/やり直す（Undo/Redo）に対応。エディタ内部の履歴スタックで管理し、ブラウザの戻る/進むとは独立させる
  - ブラウザ履歴（pushState）はメニュー/ゲーム選択やドット絵URLの共有に使っており、編集操作のUndo/Redoに転用すると競合するため
- 32x32 = 1024 セルの値（0〜63）を1セル1文字のURLセーフな64文字（base64url相当）のコードにエンコードし、URL のクエリパラメータに反映する
  - 例: `?pixels=<コード文字列>` の形式で状態を共有・復元可能にする
  - 画面上のコード表示は32文字ごとに改行し、グリッドの行に対応させる。行の高さを文字幅（1ch）に合わせることで見た目を正方形に近づける
  - ゲーム側（ライフゲーム等）は色を無視し、値が0かどうかのみで on/off を判定する
- URL を読み込むことで、そのドット絵を初期状態として編集・プレビューできる
- エディタで作成したドット絵は、各ゲームの初期パターン（例: ライフゲームの初期セル配置）や、タイトル画面のロゴ表示などに利用する
  - タイトル画面（メニュー）のロゴは `src/lib/title-logo.ts` に固定コードとして保存し、`TitleCarousel` がメニュー表示時にFavicon・ページ内プレビューへ反映する

## ゲームバリエーション

共通の 32x32 グリッド・共通入力（十字＋1ボタン）の制約の中で実装する。それぞれのゲーム画面はメニューから切り替えて選択できるようにする。

- **テトリス**: 32x32 に収まるようフィールドサイズを縮小したミニ版。フィールドは論理グリッド10x16マスで、右側にネクストミノ表示スペースを確保するため左寄せに配置（左端1マス余白、フィールド、1マスの隙間、右端4マスのネクストミノ表示枠）
  - 操作: 左右で移動、上で回転、下でソフトドロップ（落下を加速）、決定でハードドロップ（即着地）
  - フィールドの左右端には白い1px幅の境界線を描画し、盤面の範囲をわかりやすくする
  - 各ミノは種類ごとに色分け（I=シアン、O=黄、T=紫、S=緑、Z=赤、J=青、L=オレンジ）し、着地後もその色のまま固定表示する。次に出現するミノは右側の枠内にプレビュー表示する
  - ミノは4x4の枠内で4種の回転パターンを持つテーブルとして定義（壁蹴りなし。回転先が衝突する場合は回転をキャンセル）
  - ラインが揃うと消去され、スコア（消したライン数）が加算される
  - 新しいミノの出現位置が既存の積み上がりと衝突するとゲームオーバー
- **スネークゲーム**: 自機が伸びていく古典的なスネーク
- **ライフゲーム（Conway's Game of Life）**: 初期配置はデフォルトでランダム生成。ドット絵エディタで作成した初期パターンを流し込んで観察することも可能
- **フラッピーバード**: 1ボタンで上昇、常時下降する重力型アクション。自機はオレンジ、パイプ（壁）は緑で表示。難易度調整のため隙間（GAP_SIZE）は論理グリッド16マス中7マスと広めに設定
  - 自機は2x2キャラクターのままだが、パイプ・地面のスクロールは物理ピクセル1px単位の滑らかな移動にしている（キャラクターの当たり判定・描画は2x2ブロック基準のまま）
  - 死亡時は自機をその場（画面内にクランプした位置）に留め、消えないようにする。パイプ等と重なっていても自機が最前面に描画される
  - 天井に当たっても死なない。上端より上は画面外の空間として扱い、その間は自機を描画しない（突き抜けて見えなくなり、落ちてくると再び画面内に現れる）。上方向への移動量には軽い上限あり
  - 最下部には薄茶色の地面を描画。4ドットごとに色を切り替えた縞模様で、パイプと同じ速さでスクロールして動きが分かるようにする。地面に着地すると死亡
- **ぷよぷよ**: 落下パズル。フィールドは論理グリッド6x16マスで、テトリスより幅が狭いため中央寄せに配置し、右側の余白にネクストぷよ（2個組）のプレビューを表示する
  - 操作: 左右で移動、上で回転（4方向を巡回、壁蹴りなし）、下でソフトドロップ、決定でハードドロップ
  - 4色（赤・青・黄・緑）のぷよが2個1組で落下し、同色が上下左右に4個以上つながると消去される
  - ぷよを設置すると、まず浮いたぷよを重力で下に詰め、それから同色4つ以上のかたまりを判定・消去する処理を繰り返す（連鎖）。連鎖1回ごとに消去したぷよの数×連鎖数がスコアに加算される
  - 消去対象のかたまりは即座に消えるのではなく、一定時間白く点滅してから消える。連鎖が続く場合も全体が同時に消えるのではなく、1回の連鎖ごとに順番に点滅→消去が発生する
  - フィールドの左右端には白い1px幅の境界線を描画する（テトリスと同様）
  - 新しいぷよの出現位置が既存の積み上がりと衝突するとゲームオーバー
- 上記以外のゲーム追加も想定し、「ゲーム」を差し替え可能なモジュールとして扱う設計にする

## ゲーム切り替え

- タイトル画面では左右キー（十字キーの左右、Mobile は画面上の十字ボタン）でゲームを選択できる。「タイトル ↔ ゲーム1 ↔ ゲーム2 ↔ … ↔ タイトル」のようにループし、決定ボタンで選択中のゲームを開始する
  - Mobile でもタイトル画面から画面上のゲームパッド（十字＋1ボタン）を表示し、タップだけでゲーム選択・決定ができるようにする
  - ゲームの選択・切り替えはこのカルーセルのみで行い、別途ゲーム一覧のメニューは表示しない
  - 選択状態は URL に反映し、特定のゲームへ直接リンクできるようにする
  - 選択中の項目（タイトル or 各ゲーム）に応じて、Faviconおよび擬似ブラウザUIのタブ・最大化プレビューの表示も切り替わる
  - 各ゲームのプレビューは、そのゲームの初期状態を1フレーム分レンダリングしたものを暫定的に使用する（専用のアートに差し替え予定）
  - 左右に移動できることを示すページャ的な矢印マークを、Favicon画像の左右端に常時オーバーレイ表示する（ループ移動のためどちらの方向にも常に移動可能）
  - ゲーム名はタブタイトルに表示されるため、ページ内には見出しとして表示しない

## タブタイトルの動的表示

Favicon だけでなく、ブラウザタブのタイトル（`document.title`）もゲーム状態に応じて動的に書き換える。

- タイトル画面: `SUPER-FAVICON`
- ゲーム選択中（カルーセルでハイライト時）: 選択中のゲーム名のみ（例: `フラッピーバード`）
- ゲームプレイ中: `[スコア] ゲーム名` の形式でスコアを表示し、一定間隔（250ms）で更新する（例: `[100] フラッピーバード`）
  - 各ゲームは `GameInstance.getScore?: () => number` を任意で実装し、スコアの意味はゲームごとに定義する
    - フラッピーバード: 通過したパイプの本数
    - スネークゲーム: 食べたエサの数
    - ライフゲーム: 経過世代数（スコアの概念がないため代替指標として使用）
    - テトリス: 消したライン数
    - ぷよぷよ: 消去したぷよの数×連鎖数の合計
  - `getScore` を実装しないゲームはスコア0として扱う

## 開発フェーズ

構想段階から実装・公開までの開発フェーズ。各フェーズ完了時にドキュメント（本ファイル/AGENTS.md）を更新し、まとまった単位でプルリクエストを作成する。

ライフゲーム・スネークゲームの2本ができた時点で一度 Mobile 対応まで含めて公開（v1）し、残りのゲームはその後追加していく。

### Phase 0: 技術選定・環境構築
- 技術スタック: Vite + TypeScript + React、lint に oxlint を使用
- デプロイ先: Cloudflare Pages を想定
- リポジトリの雛形（ディレクトリ構成、lint/format、CI）を用意する

### Phase 1: Favicon 描画エンジン
- 32x32 ピクセルバッファ → Canvas 描画 → Favicon (`<link rel="icon">`) 更新までの最小パイプラインを実装する
- Favicon 更新頻度の上限や、ブラウザ間の挙動差を検証する
- Mobile 向けのページ内プレビュー表示（同じピクセルバッファを描画）を実装する

### Phase 2: 共通入力レイヤー
- PC（カーソルキー / WASD / Space / Enter）と Mobile（画面上の十字＋1ボタン）を、共通の「上/下/左/右/決定」イベントに正規化する入力レイヤーを実装する

### Phase 3: ゲームフレームワーク（ゲーム切り替え基盤）
- 「ゲーム」を差し替え可能なモジュールとして扱うインターフェース（初期化・毎フレーム更新・入力受付・終了判定）を設計する
- メニュー画面からのゲーム選択、URL への選択状態の反映を実装する

### Phase 4: ライフゲーム実装
- Conway's Game of Life のルールを実装
- 初期セル配置はこの時点では固定パターンをいくつか用意する（ドット絵エディタとの連携は Phase 8 で対応）

### Phase 5: スネークゲーム実装
- 自機が伸びていく古典的なスネークゲームを実装

### Phase 6: Mobile 対応・レスポンシブ調整（v1向け）
- 画面上の十字＋1ボタン UI のレイアウト調整
- ライフゲーム・スネークゲームの Mobile 実機での動作確認

### Phase 7: v1 公開
- パフォーマンス最適化（Favicon 更新頻度、描画負荷）
- 全体の動作確認、ドキュメント整備
- ライフゲーム・スネークゲームの2本で先行公開（デプロイ）

### Phase 8: ドット絵エディタ
- 32x32 グリッドへのマウス/タッチ描画 UI を実装する
- on/off ビット列 ⇔ 16進数文字列の相互変換、URL パラメータへの反映・復元を実装する
- ライフゲームの初期セル配置をエディタ経由で指定できるよう連携させる

### Phase 9: フラッピーバード実装
- 1ボタンで上昇、常時下降する重力型アクションを実装

### Phase 10: テトリス実装
- 32x32 に収まるようフィールドサイズを縮小したミニ版（例: 幅8〜10マス）を実装

### Phase 11: ぷよぷよ実装
- 落下パズルを実装。32x32 という解像度内でのフィールド設計を行う

### Phase 12: 追加ゲームの Mobile 対応確認
- フラッピーバード・テトリス・ぷよぷよの Mobile 実機での動作確認、レイアウト調整

### Phase 13: 仕上げ・最終公開準備
- 全体の動作確認、ドキュメント整備
- 全ゲーム揃った状態で公開（デプロイ）

各フェーズの完了基準や仕様の詳細は、実装を進めながら本ファイルに追記・更新していく。
